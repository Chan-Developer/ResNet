# 植物病害识别系统 设计说明书（毕业论文格式）

本文档基于仓库当前实现整理，覆盖项目结构、数据库设计与概要设计，并补充 ER 图与接口时序图。若后续代码调整，请同步更新本文件。

## 摘要

本系统面向植物病害图像识别场景，采用前后端分离架构，集成 ResNet50 模型，实现用户登录、图像识别、历史记录与数据集浏览。后端以 FastAPI 与异步 SQLAlchemy 构建服务，数据库选用 MySQL，前端采用 Vue3 + Vite + TypeScript，整体具备良好的可扩展性与工程化结构。

关键词：植物病害识别，ResNet50，FastAPI，Vue3，MySQL

## 第 1 章 绪论

### 1.1 研究背景

植物病害诊断依赖人工经验，识别成本高且易受主观影响。利用深度学习模型进行图像识别，能够有效提升诊断效率与一致性。

### 1.2 系统目标

系统目标包括：
- 实现植物病害图像的在线识别
- 支持用户登录、识别记录管理与查询
- 支持数据集类别与样本浏览
- 提供可扩展的后端服务与前端交互界面

### 1.3 技术选型

技术选型如下：
- 后端：FastAPI + SQLAlchemy 2.0（异步）+ MySQL + JWT
- 前端：Vue3 + Vite + TypeScript + Element Plus + Pinia
- 模型：ResNet50（38 类）

### 1.4 运行环境

运行环境要求：
- Python 3.10+（建议）
- Node.js 16+（建议）
- MySQL 8.0+（建议）

1. 前端上传图片文件

1. 前端请求历史列表
2. 后端按用户分页查询
3. 返回分页数据

1. 前端请求类别列表
2. 后端读取数据集目录并统计数量
3. 按类别分页返回图片

## 第 6 章 扩展需求设计（RAG + 复查 + 预警 + 统计）

本章针对新增需求给出完整的功能与数据设计，作为后续开发的实现依据。设计以“可落库、可追踪、可验证”为原则。

### 6.1 需求一：诊断上传流程（RAG + LLM）

目标：在现有识别基础上，补充知识片段检索与相似病例检索，并由 LLM 生成可引用、有不确定性提示的处置建议。经用户确认后建档入库并推荐相似病例与参考方案。

流程定义（高层）：  
用户上传叶片图片  
→ 图像预处理  
→ 模型识别病害类别与健康状态  
→ 检索知识片段与相似病例  
→ 基于 RAG 的 LLM 生成处置建议（含引用与不确定性提示）  
→ 用户确认并建档  
→ 保存数据库  
→ 推荐相似病例与参考处理方案

关键设计点：  
- 知识片段与病例检索结果必须可追溯（记录检索来源、评分）。  
- LLM 输出必须包含引用与“不确定性提示”。  
- “建档”必须经过用户确认，避免误记录。  
- 相似病例推荐与处置方案引用需来源化展示。

建议新增组件：  
- 向量检索服务（知识片段与病例嵌入检索）  
- LLM 服务（RAG Prompt 模板 + 引用约束）  
- 病例管理服务（确认建档、关联历史预测）

建议新增数据结构（核心字段示例）：  
- `knowledge_chunk`  
  - `id`, `source_type`, `source_name`, `content`, `tags`, `embedding_id`, `updated_at`  
- `disease_case`  
  - `id`, `user_id`, `prediction_record_id`, `confirmed_label`, `status`, `location_lat`, `location_lng`, `region_code`, `created_at`, `updated_at`  
- `case_evidence`  
  - `id`, `case_id`, `evidence_type(knowledge|case)`, `title`, `url`, `snippet`, `score`

建议新增接口（示意）：  
- `POST /api/case/confirm`（确认建档）  
- `GET /api/case/{id}`（病例详情）  
- `GET /api/case/{id}/similar`（相似病例）  
- `GET /api/knowledge/search`（知识片段检索）

### 6.2 需求二：病情复查与跟踪

目标：支持用户设置复查计划，定期上传复查图片，系统对比历史特征与置信度变化，生成变化说明与建议，并输出趋势图与效果评估。

流程定义（高层）：  
用户设置复查计划  
→ 定期上传复查图片  
→ 比较历史与当前特征/置信度变化  
→ 检索变化相关证据  
→ LLM 生成变化说明与建议  
→ 更新病情状态  
→ 输出趋势图与处理效果评估

关键设计点：  
- 复查计划必须绑定病例。  
- 对比指标包含 Top1 类别变化、置信度变化、相似度变化（可用嵌入向量）。
